# Commax Call 에이전트 작업 지침

이 파일은 저장소 전체에 적용됩니다. Codex를 포함한 AI 에이전트는 작업을 시작하기 전에 이 파일과 `README.md`를 읽고, 사용자의 최신 요청을 최우선으로 따릅니다.

## 최우선 원칙: 최신 Home Assistant 호환성

이 프로젝트는 **현재 최신 안정판 Home Assistant와의 호환성**을 가장 중요한 기준으로 삼습니다. Home Assistant는 릴리스와 API 변경이 빠르므로, 과거 버전에서 동작했다는 사실만으로 현재도 올바른 구현이라고 판단하지 않습니다.

- 사용자가 특정 버전을 지정하지 않는 한 최신 안정판 Home Assistant를 기본 지원 대상으로 합니다.
- 최신 안정판과 과거 버전의 구현 방식이 충돌하면 최신 안정판의 공개 API와 권장 방식을 우선합니다.
- Home Assistant API와 수명 주기 메서드를 변경할 때 모델의 기억, 오래된 블로그, 과거 예제 코드만 근거로 사용하지 않습니다.
- 변경 전에 Home Assistant 공식 개발 문서, 공식 개발자 블로그의 deprecation 공지, `home-assistant/core` 저장소 `dev` 브랜치의 현재 구현과 테스트를 확인합니다.
- 안정판 문서는 현재 호환성의 기준으로 사용하고, Core `dev` 브랜치는 다음 릴리스에서 발생할 변경을 미리 파악하는 참고 자료로 사용합니다.
- deprecated API에 공식 대체 API가 있으면 새 코드에서 deprecated API를 사용하지 않습니다. 기존 코드에서 발견하면 해당 작업과 직접 관련된 범위에서 안전하게 교체합니다.
- 제거된 API를 호환성 우회 코드로 되살리거나, 버전별 조건문을 무분별하게 추가하지 않습니다. 정말 여러 HA 버전을 지원해야 할 때만 지원 범위와 이유를 문서화합니다.
- Home Assistant 내부 구현에 해당하는 비공개 속성이나 밑줄 접두사 API에 의존하지 않고, 공개된 통합 구성요소 API를 사용합니다.
- 최신 HA가 요구하는 Python 버전, 비동기 패턴, 타입 표기, Config Flow, Config Entry, 엔티티 수명 주기 및 manifest 형식을 함께 확인합니다. 버전 번호나 요구사항을 기억에 의존해 하드코딩하지 않습니다.
- 수정 후 Home Assistant 로그에 새 deprecation 경고가 발생하지 않는지 확인합니다. 실제 HA 실행 환경을 사용할 수 없다면 검증하지 못한 항목으로 명확히 보고합니다.

### HA API 변경 작업 절차

Home Assistant API와 맞닿은 코드를 수정할 때 다음 순서를 따릅니다.

1. 현재 프로젝트가 사용하는 클래스, 함수, 상수와 호출 위치를 검색합니다.
2. 최신 공식 개발 문서에서 현재 권장 API와 함수 시그니처를 확인합니다.
3. 필요하면 `home-assistant/core`의 `dev` 브랜치에서 같은 API를 사용하는 최신 통합 구성요소와 테스트를 확인합니다.
4. 공식 deprecation 또는 제거 공지가 있는지 개발자 블로그와 릴리스 관련 문서를 확인합니다.
5. 관련 없는 전면 리팩터링을 피하고 호환성 수정 단위를 작게 유지합니다.
6. 가능한 경우 최신 Home Assistant 테스트 환경에서 설정, 로딩, 언로드, 옵션 갱신과 엔티티 상태 변경을 검증합니다.
7. 확인한 HA 기준 버전 또는 확인 날짜, 참고한 공식 자료, 남은 호환성 위험을 결과에 기록합니다.

### 지원 범위

- 최신 안정판 Home Assistant에서 정상 동작하는 것이 필수 목표입니다.
- 바로 이전 버전과의 호환성은 구현 복잡도와 안전성을 해치지 않는 범위에서 유지할 수 있지만 필수 목표는 아닙니다.
- 오래된 버전을 위한 코드가 최신 API 사용을 방해하면 제거를 검토하고, 사용자 영향과 필요한 최소 HA 버전을 `README.md` 또는 릴리스 안내에 명시합니다.
- 최신 안정판에서 직접 시험하지 못한 경우 “최신 버전 호환 완료”라고 단정하지 않습니다.

## 프로젝트 목표

Commax 월패드 통신 패킷을 EW11 TCP 게이트웨이를 통해 송수신하고, Home Assistant에서 벨 바이너리 센서와 명령 스위치로 사용할 수 있게 합니다.

우선순위는 다음과 같습니다.

1. 최신 안정판 Home Assistant와의 API 및 수명 주기 호환성
2. 호출 감지와 패킷 송신의 신뢰성
3. Home Assistant 이벤트 루프 및 스레드 안전성
4. 기존 사용자의 Config Entry와 엔티티 ID 호환성
5. 연결 장애 후 자동 복구
6. 사용자가 이해할 수 있는 한글 문서와 주석

## 작업 전 확인

- `git status --short --branch`로 사용자의 미커밋 변경사항을 확인하고 보존합니다.
- 전체 구조는 `README.md`를 먼저 읽고 파악합니다.
- 변경 대상과 직접 연결된 파일을 함께 읽습니다. 예를 들어 패킷 수신 변경은 `hub.py`, `binary_sensor.py`, `const.py`를 함께 확인합니다.
- Home Assistant API를 건드리는 작업은 최신 공식 자료를 인터넷에서 확인합니다. 검색 결과의 요약만 믿지 말고 원문을 확인합니다.
- 현재 동작을 확인하지 않은 상태에서 예제 코드나 일반적인 Home Assistant 패턴으로 기존 로직을 대체하지 않습니다.
- 버그 수정은 가능한 경우 재현 조건과 원인을 먼저 찾은 뒤 최소 범위로 수정합니다.

## 핵심 구조와 불변조건

- `__init__.py`는 Config Entry별 `Hub`를 생성하고 `switch`, `binary_sensor` 플랫폼의 수명 주기를 관리합니다.
- `hub.py`는 EW11 연결, 재연결, TCP 스트림 버퍼링, 패킷 송수신을 담당합니다.
- `binary_sensor.py`는 완성된 수신 패킷을 설정값과 비교하여 벨 상태를 바꿉니다.
- `switch.py`는 사용자의 스위치 조작을 HEX 패킷 송신으로 변환합니다.
- `config_flow.py`와 `translations/`는 최초 설정 및 옵션 UI를 함께 구성합니다.
- 현재 Commax 프레임은 16바이트, 시작 `0x02`, 종료 `0x03`을 전제로 합니다. 프로토콜 근거 없이 이 값을 변경하지 않습니다.
- TCP의 `recv()` 호출 단위는 패킷 경계가 아닙니다. 일부 패킷 또는 여러 패킷이 한 번에 도착할 수 있으므로 `_extract_packets()`의 누적 버퍼 방식을 유지합니다.
- 블로킹 소켓 I/O를 Home Assistant의 asyncio 이벤트 루프에서 직접 실행하지 않습니다.
- 소켓 교체, 종료, 송신이 동시에 일어날 수 있으므로 `_socket_lock` 보호 범위를 약화하지 않습니다.
- EW11 연결이 끊기면 새 소켓으로 재연결해야 하며, 통합 구성요소 언로드 중에는 재연결하지 않아야 합니다.

## Home Assistant 호환성

- `DOMAIN`은 디렉터리 이름과 `manifest.json`의 `domain`인 `commax_call`과 일치해야 합니다.
- `const.py`의 `CONF_*` 문자열은 Home Assistant에 영구 저장되는 데이터 키입니다. 키를 변경해야 한다면 기존 Config Entry를 위한 마이그레이션도 함께 구현합니다.
- 기존 엔티티의 `unique_id`, 장치 `identifiers`, Config Entry ID 조합을 특별한 이유 없이 변경하지 않습니다. 변경하면 사용자의 엔티티 설정과 자동화가 끊어질 수 있습니다.
- 옵션을 저장한 뒤 Config Entry가 다시 로딩되는 흐름을 유지합니다.
- 설정 필드를 추가하거나 이름을 바꾸면 `config_flow.py`, `const.py`, `translations/ko.json`, `translations/en.json`, `README.md`를 함께 갱신합니다.
- Home Assistant API를 호출할 때 현재 지원 버전의 비동기 API와 엔티티 수명 주기를 따릅니다.
- 같은 저장소의 오래된 코드보다 최신 Home Assistant 공식 문서와 Core 구현을 호환성 판단의 기준으로 삼습니다.
- 공식 API가 바뀌었을 때 호환성 어댑터가 필요한지, 최신 API로 완전히 전환할지를 지원 범위에 따라 결정합니다.
- 백그라운드 수신 스레드나 `threading.Timer`에서 HA 상태를 갱신할 때는 이벤트 루프에 안전하게 전달되는지 확인합니다.

## 패킷 처리 원칙

- 사용자가 입력하는 패킷은 공백으로 구분된 HEX 문자열이며 `bytes.fromhex()` 또는 `bytearray.fromhex()`로 변환됩니다.
- 잘못된 HEX, 빈 패킷, 길이가 다른 패킷 때문에 통합 구성요소 전체가 중단되지 않도록 입력 검증과 오류 처리를 고려합니다.
- 패킷 비교는 바이트 단위로 수행하고, 단순 문자열 비교로 바꾸지 않습니다.
- 송신에는 전체 데이터가 전달되도록 `socket.sendall()`을 사용합니다.
- 실제 IP 주소와 통신 패킷은 민감한 환경 정보일 수 있습니다. 일반 로그에 과도하게 남기지 말고 디버그 로그에서도 외부 공유 가능성을 고려합니다.
- 실제 하드웨어 동작을 추정해서 패킷 값을 새로 만들지 않습니다. 패킷 의미가 불명확하면 사용자에게 캡처 데이터나 장치 모델을 요청합니다.

## 코드 작성 규칙

- 코드 식별자는 Python 및 Home Assistant 관례에 맞춰 영어를 사용합니다.
- 주석, docstring, 사용자용 문서는 사용자의 이해를 위해 한글로 작성합니다. 고유 API 명칭과 코드 식별자는 그대로 표기할 수 있습니다.
- 주석은 코드 문장을 그대로 번역하기보다 설계 이유, 스레드 경계, 패킷 조건처럼 코드만 보고 알기 어려운 내용을 설명합니다.
- 새 코드에는 명확한 이름과 작은 책임 단위를 사용합니다. 관련 없는 대규모 포매팅이나 구조 변경을 한 작업에 섞지 않습니다.
- 기존 공개 동작을 바꾸는 리팩터링은 테스트 또는 명확한 검증 근거와 함께 진행합니다.
- 새 의존성은 꼭 필요한 경우에만 추가하고 `manifest.json`의 `requirements` 및 설치 영향을 함께 검토합니다.
- `manifest.json`과 `const.py`의 버전 값은 서로 일치시킵니다. 단순 로컬 수정만으로 버전을 임의 변경하지 말고 릴리스 작업에서 함께 변경합니다.

## 알려진 기술 부채

다음 항목은 관련 작업을 할 때 우선 확인하되, 사용자 요청과 무관한 수정에 섞지 않습니다.

- `binary_sensor.py`의 `Device`와 `device.py`의 `Device` 역할이 중복됩니다.
- `Device.publish_updates()` 및 `BellSensor.set_state()`에 중복 정의가 있습니다.
- 사용하지 않는 import와 실행되지 않는 빈 메서드가 남아 있습니다.
- `hass.data[DOMAIN]["listener"]`는 여러 Config Entry가 있을 때 공유 상태가 될 가능성이 있습니다.
- 일부 엔티티 상태 갱신이 소켓 수신 스레드 또는 타이머 스레드에서 시작됩니다.
- `Hub.hub_id`가 참조하는 `_id`는 현재 생성자에서 초기화되지 않습니다.
- 엔티티의 `available` 값이 실제 Hub 연결 상태를 반영하지 않습니다.

기술 부채를 수정할 때는 한 번에 하나의 위험을 다루고 기존 동작을 보존하는 테스트를 먼저 추가하는 방식을 선호합니다.

## 검증 방법

변경 범위에 맞는 검증을 수행합니다. 최소 정적 검증은 다음과 같습니다.

```bash
python3 -m compileall -q .
python3 -m json.tool manifest.json >/dev/null
python3 -m json.tool translations/ko.json >/dev/null
python3 -m json.tool translations/en.json >/dev/null
git diff --check
```

패킷 조립 또는 상태 전이 로직을 변경할 때는 가능하면 다음 경우를 자동 테스트합니다.

- 한 패킷이 여러 TCP 조각으로 나뉘어 수신되는 경우
- 여러 패킷이 한 번에 수신되는 경우
- 패킷 앞에 불필요한 바이트가 포함된 경우
- 시작/종료 바이트 또는 길이가 잘못된 경우
- 연결 종료 후 재연결되는 경우
- 벨 시작, 벨 종료, 통화 종료, 자동 OFF 타이머
- 스위치 ON/OFF 패킷과 자동 OFF

실제 EW11이나 Commax 장치가 없는 환경에서는 하드웨어 검증을 했다고 주장하지 않습니다. 정적 검사, 단위 테스트, 모의 소켓 테스트와 실제 장치 확인 필요 항목을 구분하여 보고합니다.

Home Assistant 연동 API를 변경했다면 정적 검사만으로 완료 처리하지 않습니다. 가능한 경우 최신 안정판 HA가 설치된 테스트 환경에서 다음 항목을 확인합니다.

- 통합 구성요소 추가와 Config Flow 완료
- Home Assistant 시작 시 Config Entry 로딩
- 옵션 변경 후 정상 리로드
- Home Assistant 재시작 후 설정과 엔티티 유지
- 통합 구성요소 삭제 또는 언로드
- deprecation 경고와 이벤트 루프 블로킹 경고가 발생하지 않음
- 엔티티 및 장치 레지스트리에 중복 항목이 생성되지 않음

최신 HA 실행 환경이 없다면 공식 문서 및 Core 코드 확인 결과와 정적 검사까지만 완료했다고 보고하고, 실제 최신 안정판 검증이 남았음을 명시합니다.

- 사용자에게 변경 목적, 중요한 파일, 수행한 검증, 확인하지 못한 하드웨어 항목을 한글로 간결하게 보고합니다.
- Home Assistant API 관련 변경은 확인한 공식 문서, 대상 HA 버전 또는 확인 날짜를 함께 보고합니다.
- 발견한 문제와 실제 수정한 문제를 구분합니다.
- 요청받지 않은 커밋, 푸시, 릴리스 또는 버전 태그 생성을 하지 않습니다.
- 커밋이나 푸시를 요청받으면 먼저 변경 파일과 검증 결과를 확인하고 의미가 분명한 커밋 메시지를 사용합니다.

## 완료 기준

- 요청한 동작 또는 문서가 구현되었습니다.
- 기존 미커밋 작업이 보존되었습니다.
- 관련 정적 검사와 가능한 테스트를 통과했습니다.
- Home Assistant API 변경이 있다면 최신 안정판 공식 문서와 현재 Core 구현을 확인했습니다.
- Config Entry, 엔티티 ID, TCP 패킷 경계, 스레드 안전성에 미치는 영향을 검토했습니다.
- 남은 위험이나 실제 하드웨어에서 확인할 사항을 사용자에게 알렸습니다.
